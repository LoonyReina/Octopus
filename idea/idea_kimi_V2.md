# OpenClaw 模拟宠物系统改造方案（Kimi 分级版 V2）

> 基于 idea_V1 的 11 个分析模块，合并相似需求、按落地优先级重新分级，保留对 OpenClaw 的核心改造点。

---

## 一、合并后的核心改造层

将原 11 个模块合并为 **6 个核心层 + 2 个支撑层**：

| 合并后层级 | 涵盖原模块 | 核心问题 |
|-----------|-----------|---------|
| **A. 感知层** | 原 1（感知）+ 原 9（实时性）部分 | 多模态输入如何转为结构化事件 |
| **B. 身份层** | 原 2（身份建模） | 主人是谁、如何区分、档案存什么 |
| **C. 记忆层** | 原 5（记忆分层）+ 原 2 部分 | 短期/情景/语义/实体/技能记忆如何分层存储与检索 |
| **D. 技能层** | 原 3（技能）+ 原 6（教学闭环） | 动作如何表示、教学如何闭环、如何判定学会 |
| **E. 状态与决策层** | 原 4（状态/情绪）+ 原 7（决策/Policy） | 宠物内部状态如何驱动行为，如何主动互动 |
| **F. 跨模态 Grounding** | 原 8（跨模态对齐） | 指代、时间对齐、实体绑定 |
| **G. 实时工程层** | 原 9（实时性/架构） | 低/中/高延迟分层、异步化、本地优先 |
| **H. 隐私安全与评测层** | 原 10（隐私）+ 原 11（评测） | 敏感数据保护、记忆可审计、指标体系 |

---

## 二、优先级分级（保留核心，删减理想化需求）

### P0 — MVP 必须先做（没有就不叫模拟宠物）

**目标：最小可行地体现「认主人 + 被教会动作 + 有宠物感」**

1. **记忆层升级：结构化实体记忆 + 主人绑定**
   - **现状问题**：OpenClaw 的 `active-memory` 是纯文本摘要+检索，没有「这个人是谁」的概念。
   - **改造点**：
     - 在 `extensions/active-memory/` 中增加 **Entity Memory** 子模块，维护 `owner_profile` 表（称呼、偏好、简单历史互动）。
     - 每次会话启动时（`session.ts` 阶段），将主人档案注入 System Prompt，让 LLM 知道「当前用户是谁、之前聊过什么偏好」。
     - 记忆召回从「全局文本搜索」改为「身份关联搜索」：优先召回与当前用户相关的记忆。
   - **不做会怎样**：宠物对所有人一视同仁，没有「认主人」感。

2. **状态与决策层（极简版）：内部状态变量 + Prompt 注入**
   - **现状问题**：OpenClaw 是被动响应式，没有内部状态。
   - **改造点**：
     - 新增极简状态机（`mood`, `affection`, `energy` 三变量），在 `agent-runner.ts` 阶段读取并注入 System Prompt。
     - 例如：`affection=high` 时注入「你对主人很亲昵，用词活泼」；`energy=low` 时注入「你有点困，回复简短」。
     - 状态更新规则先写死：主人发消息 → affection+1；长时间不理 → energy 下降。
   - **不做会怎样**：宠物永远是同一种语气，没有情绪变化，像客服不像宠物。

3. **技能层（指令映射版）：结构化技能记录 + Prompt 召回**
   - **现状问题**：OpenClaw 的 Skills 是「外部工具包」，不是「宠物学会的本领」。
   - **改造点**：
     - 复用 `agents/skills/workspace.ts` 的技能加载机制，但新增 **Learned Skill** 类型。
     - 主人教动作时（文本指令），解析为结构化记录：`{name, trigger_word, action_desc, taught_by, success_examples[]}`，存入技能记忆。
     - 每次会话将「已学会的技能列表」注入 System Prompt，让 LLM 知道「主人教过我坐下」。
   - **不做会怎样**：主人每次说「坐下」，宠物都像是第一次听。

**P0 边界说明（明确不做）**：
- 不做真正的人脸/声纹识别（MVP 用文本 self-report 或手动绑定身份）。
- 不做视频理解（MVP 只用文本+语音转文字）。
- 不做主动发起互动（P0 仍保持被动响应，但通过状态改变回复风格）。
- 不做强化学习或真正可执行的动作控制（技能只停留在文本描述层）。

### P1 — 第二阶段：核心体验闭环（让宠物真正可用、可长期陪伴）

4. **身份层：独立 Owner Profile + 多用户区分**
   - 建立独立身份存储（`extensions/identity/` 新插件），区分主人、家庭成员、陌生人。
   - 至少实现基于文本/昵称的身份区分，后续再叠加人脸/声纹。
   - 在 `session.ts` 中增加身份识别阶段：先识别说话人，再加载对应记忆上下文。

5. **感知层（基础多模态）：视频抽帧 + 语音事件流**
   - 新增 `extensions/perception/` 插件：
     - 视频以固定间隔抽帧，调用多模态模型（或本地小模型）生成结构化事件：`{timestamp, event_type, description, involved_entities[]}`。
     - 语音除转文字外，提取语气标签（开心/生气/平静）。
   - 事件流代替原始视频进入记忆层，解决「直接喂视频给模型」的不稳定和昂贵问题。

6. **教学闭环（反馈学习）：基于多模态反馈更新技能**
   - 在 P0 的指令映射基础上，增加：
     - **纠错识别**：主人说「不对，不是这个」→ 降低该技能置信度。
     - **奖励识别**：主人说「真棒」+ 抚摸（通过感知层识别为正向反馈）→ 提高置信度。
     - **泛化记录**：同一动作在不同场景下成功，更新 `success_examples`。
   - 在 `pi-embedded-subscribe.handlers.tools.ts` 或新增 hook 中实现 `after_user_turn` 反馈分析。

7. **记忆层完整化：情景记忆 + 记忆巩固/遗忘**
   - 原 `active-memory` 只有「近期摘要」，P1 需要：
     - **情景记忆**：存储具体互动片段（如「2025-04-24 下午在客厅教握手」）。
     - **语义记忆**：提取抽象知识（如「主人喜欢狗」）。
     - **巩固机制**：每天离线总结情景记忆→语义记忆，压缩存储。
     - **遗忘机制**：低频记忆衰减，重要性评分低的被覆盖。

### P2 — 第三阶段：高阶像真（让它更像真宠物）

8. **状态与决策层（完整版）：主动行为 + Policy 层**
   - 从被动响应升级为 **事件驱动 + 状态驱动混合架构**：
     - 状态层持续运行（`energy` 随时间自然恢复/消耗，`mood` 受环境事件影响）。
     - Policy 层独立决策：当 `boredom > threshold` 时，主动向主人发送消息；当检测到主人靠近时，主动展示亲热行为。
   - 需要改造 `auto-reply/reply/get-reply.ts` 的入口：不仅处理用户输入，还要处理内部状态触发的事件。

9. **跨模态 Grounding：指代消解与时间对齐**
   - 解决：「把刚才那个动作再做一遍」、「上次在客厅教你的你还记得吗」。
   - 需要：
     - 指代消解模块：解析「那个动作」→ 映射到具体技能 ID。
     - 时空检索：在情景记忆中按 `{location, time_range, action_type}` 检索。
     - 跨模态对齐：将视频片段的时间戳与对话轮次对齐。

10. **技能层（执行版）：从文本描述到可执行输出**
    - 技能不再只是 Prompt 里的描述，而是真正触发外部动作：
      - 控制硬件（如机器人、智能家居）。
      - 控制虚拟形象动画/音效。
    - 需要扩展 `plugin-sdk` 的 `SkillProvider` 接口，支持 `execute()` 回调。

---

### P3 — 工程支撑（生产级要求）

11. **实时工程分层**
    - **低延迟层（边缘本地）**：唤醒检测、主人身份快速匹配、简单状态响应（如检测到主人回家立即摇尾巴）。
    - **中延迟层（网关）**：语音理解、视频事件解析、技能执行。
    - **高延迟层（异步后台）**：记忆巩固、个性更新、长期趋势分析。
    - OpenClaw 当前架构是「请求-响应」串行，P3 需要引入异步管道和边缘计算节点。

12. **隐私安全**
    - 人脸/声纹特征本地加密存储，不上传原始视频。
    - 记忆可查看、可删除、可纠正（UI 提供审计面板）。
    - 身份误认风险控制：陌生人置信度低于阈值时，默认标记为 `unknown`，禁止访问主人私有记忆。

13. **评测体系**
    - MVP 优先监控：主人识别准确率（文本层）、技能召回率（教过的动作能否答出）、状态一致性（同一情境下情绪是否稳定）。
    - 架构健康指标：记忆检索延迟、多模态事件丢失率、主动行为误触发率。

## 三、对 OpenClaw 核心文件的具体改造映射

| OpenClaw 核心文件 | 当前生态位 | 模拟宠物改造角色 |
|-------------------|-----------|-----------------|
| `src/entry.ts` | 进程入口 | 增加宠物模式启动参数 `--pet-mode`，加载宠物专用插件集 |
| `src/auto-reply/reply/get-reply.ts` | 消息路由/调度 | 改造为**事件总线**：不仅处理用户消息，还处理状态触发事件、感知事件 |

| `src/agents/pi-embedded-runner/run.ts` | LLM 运行时 | 增加 `after_user_turn` hook，用于状态更新和技能反馈分析 |

| `src/plugins/memory-state.ts` | 记忆状态 | 扩展为分层记忆接口（实体/情景/语义/技能） |

| `ui/src/ui/controllers/skills.ts` | 技能管理 UI | 新增「已学会动作」管理、教学记录查看 |
| `ui/src/ui/views/agents.ts` | Agent 面板 | 新增「宠物状态监控」卡片（心情、精力、亲密度） |
| `extensions/active-memory/` | 文本记忆插件 | 核心重构：多模态、分层、身份关联、巩固遗忘 |

## 四、关键判断题（直接表态）

1. **「记住主人」更像记忆问题，还是身份建模问题？**
   - **身份建模问题**。普通 RAG 记忆是「内容检索」，记住主人是「实体绑定+关系建模」。OpenClaw 当前没有 entity store，必须在记忆层之上新增身份层。

2. **「记住主人教过的动作」更像记忆问题，还是技能学习问题？**
   - **技能学习问题**。如果只是把「主人教过转圈」写进记忆，宠物只能复述这句话；技能学习需要「触发条件 → 执行逻辑 → 成功判定 → 泛化映射」的完整表示，是可执行的结构化知识。

3. **「像宠物一样互动」更依赖多模态记忆，还是状态机/行为策略？**
   - **状态机/行为策略**。多模态记忆让宠物「知道过去发生了什么」，但「宠物感」来自当前状态和决策：同样的历史，心情好和心情不好时回应完全不同。没有状态层，宠物只是记忆丰富的客服。

4. **如果只把视频理解 + 多模态 RAG 做好，离真正的模拟宠物还差哪些关键能力？**
   - 还差：**身份建模**（认人）、**技能学习**（学会动作而不仅是记住描述）、**状态与 Policy**（情绪驱动的主动行为）、**教学闭环**（从被教到学会的系统流程）。

5. **如果只允许选 3 个最关键改造方向，选哪 3 个？**
   - **① 结构化实体记忆 + 主人绑定**（解决认主人）
   - **② 极简内部状态 + Prompt 注入**（解决宠物感）
   - **③ 可学习的技能记录 + 教学闭环**（解决被教会动作）
   - 理由：这三个分别对应 idea_V1 最核心的三个用户诉求，且都能在现有 OpenClaw 架构内以最小改造实现 MVP。

## 五、最务实的 MVP 架构（P0 实现版）

```
[用户输入] ──→ [get-reply.ts 事件总线]
                     │
                     ▼
         ┌─────────────────────┐
         │   Session Init      │ ← 加载 owner_profile + pet_state + learned_skills
         │   (session.ts)      │
         └─────────────────────┘
                     │
                     ▼
         ┌─────────────────────┐
         │   Prompt Assembler  │ ← System Prompt = 原提示 + 主人档案 + 宠物状态 + 技能列表
         │ (get-reply-run.ts)  │
         └─────────────────────┘
                     │
                     ▼
         ┌─────────────────────┐
         │   LLM Runner        │ ← 正常对话，但支持技能触发词识别
         │ (pi-embedded-runner)│
         └─────────────────────┘
                     │
                     ▼
         ┌─────────────────────┐
         │   After-Turn Hook   │ ← 更新 affection/energy，分析是否涉及教学/纠错
         │ (handlers/tools.ts) │
         └─────────────────────┘
                     │
                     ▼
              [保存记忆 + 状态]
```

**MVP 明确不做**：
- 人脸识别/声纹识别（用文本身份绑定代替）
- 视频理解（纯文本+语音文字）
- 主动发起互动（保持被动响应）
- 真正的动作执行（技能停留在文本描述）
- 强化学习（规则驱动的置信度更新）
- 跨模态 Grounding（只处理明确指代）

---

*V2 设计原则：在 OpenClaw 现有插件化架构上做最小侵入式改造，优先利用 `active-memory`、`skills/workspace.ts`、`session.ts`、`agent-runner.ts` 的现有钩子，而不是推翻重写。*
